# 카카오 간편가입(카카오싱크) 도입 체크리스트

간편 로그인 → 간편 회원가입(카카오싱크) 전환을 위한 **카카오 쪽 선행 작업 + 우리 쪽 남은 배선** 정리.
근거는 카카오 비즈니스/디벨로퍼 공식 문서와 데브톡(2026-07 조사, 컷오프 이후일 수 있으니 신청 전 재확인).

## 지금 코드가 이미 하는 것 (검수 불필요)

- 카카오 로그인 자체는 `phone_login_page`의 `_oauthOptions`로 렌더, `Env.enabledOAuth`(`ENABLED_OAUTH`)에 `kakao` 포함 시 활성. **현재 기본값은 `google`뿐이라 카카오 버튼은 미노출.**
- `auth.dart`가 카카오에 `profile_nickname` scope만 요청(이메일 동의항목 미설정 에러 회피).
- **0032**: 로그인 직후 `_postLoginNav`가 카카오 닉네임을 `capture_oauth_profile`로 `display_name`에 자동 반영 → 온보딩 이름 재입력 제거. (기존 이름은 안 덮음. 편의용 표시명일 뿐 실명 아님)
- 자체 약관 동의 인프라(`consents`/`record_consents`, 필수 5종)는 이미 있음 → 카카오싱크의 약관 일괄동의와 **중복**되므로, 카카오싱크 도입 시 어느 쪽을 정본으로 할지 결정 필요.

## 카카오 쪽 선행 작업 (사용자만 가능 — 순서대로)

1. **비즈니스앱(비즈앱) 전환** — 카카오 디벨로퍼스 콘솔. 앱 이름·앱ID·OWNER 계정 본인인증·전환 목적 제출 후 검수. 개인/테스트 앱으로는 카카오싱크 불가.
2. **비즈니스용 카카오톡 채널 생성 + 앱 연결** — 카카오싱크의 전제.
3. **카카오싱크 신청** — 비즈앱 + 채널 연결 후 활성화. 활성화되면 카카오가 띄우는 동의 화면 한 장에서 약관+개인정보 동의를 일괄 처리.
4. **개인정보 동의항목 검수** (추가기능 신청 > 각 항목별) — 아래는 전부 **비즈앱 + 검수 통과** 필요. 기본 scope로는 하나도 못 받음:
   - `account_email` (이메일)
   - `name` (이름) — 검수는 "실제 회원가입에서 그 항목을 수집·사용하는지" 확인 → 우리 앱의 회원가입/정산에서 쓰는 근거 제시
   - `phone_number` (카카오계정 전화번호) — 반려 흔함. 알림톡 발송 등 명확한 이용목적 입증 필요. 카카오톡 등록 전화번호만 제공(카카오계정 이메일가입자는 제공 불가), `+82` 국제표기로 올 수 있음
   - `birthday`/`birthyear` (생일/출생연도)
   - `account_ci` (CI) — **수집 증빙자료 사전 제출 필수**. 회원식별 용도 외 저장·활용 불가

## 카카오 CI에 대한 중요한 한계 (신뢰 판단과 직결)

- **카카오는 본인확인기관이 아니다.** `account_ci`는 카카오가 본인확인을 *수행*해 만드는 값이 아니라, 사용자가 자기 카카오계정에서 이미 휴대폰 본인확인을 완료했을 때 그때 발급돼 카카오가 보관 중인 CI를 반환하는 것뿐. **본인인증 안 한 사용자는 CI가 없다.**
- 따라서 카카오 CI는 **중복가입 식별(DI 대용) 보조수단**이지, 플랫폼의 법적 **본인확인**이 아니다. 실 본인확인(실명+생년월일+CI/DI 확정)은 여전히 **본인확인기관 연동(0031 `apply_identity_verification`)**이 정본 — 카카오싱크는 이걸 대체하지 못한다.

## 우리 쪽 남은 배선 (카카오 검수 통과 후)

1. `auth.dart` `signInWithOAuth`의 카카오 `scopes`에 검수 통과한 항목 추가(예: `'profile_nickname account_email name phone_number'`).
2. **Supabase 주의**: OAuth scope로 받은 값은 Supabase가 `user_metadata`에 영구 저장하지 않음 — 로그인 직후 세션에서만 노출. `_postLoginNav`의 `currentUser?.userMetadata` 캡처 로직을 확장해 name/phone을 잡아 서버 RPC로 저장(현재 0032는 `display_name`만; phone/CI용 캡처 RPC 추가 필요).
3. phone을 프로필에 넣을지 결정 — 우리 OTP로 검증한 phone(`phone_verified=true`)과 카카오 제공 phone(미검증)은 구분해서 저장. 섞으면 `phone_verified` 의미가 깨짐.
4. `ENABLED_OAUTH`에 `kakao` 추가한 빌드 배포(서버 설정 완료 후).

## 판단 요약

- **간편가입 = 전환율/편의 레버**(이름·전화 재입력 제거, 약관 UX). 신뢰 문제는 **안 풀어줌**.
- **신뢰 = 본인확인기관 연동(0031)**이 정본. 둘은 별개이고, 우선순위는 신뢰(A) > 결함(B) > 카카오싱크(C).
- 지금 0032로 "검수 없이 되는 부분(닉네임 자동 표시명)"은 이미 확보. 나머지는 비즈앱 검수라는 카카오 쪽 리드타임에 종속.
